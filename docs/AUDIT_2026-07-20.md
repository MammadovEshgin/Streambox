# StreamBox Full Engineering + Design Audit — 2026-07-20

Report-only audit. No code was changed. Scope: performance, network resilience (mobile-network search/load failures), low-bandwidth behavior, code quality, architecture, and product design. Companion implementation plan at the end. Goal: top-tier mobile app quality with zero functional regressions.

---

## Executive summary

The codebase is **well above average** for an indie app: real design tokens, persisted runtime caches, in-flight request dedup, LRU-bounded memory caches, an error-boundary-guarded startup, telemetry, and 23 test files. This is not a slop codebase.

The "search returns nothing / movie won't load, works after waiting, worse on mobile networks" symptom has a **specific, provable root-cause chain**:

1. The Cloudflare Worker TMDB proxy rate-limits at **120 requests/minute per IP**.
2. The app fires **request bursts of 40–120+ proxied calls** on cold start and browsing (spotlight resolves up to 36 `/find` calls; IMDb Top-250 pages resolve 20 `/find` each; detail pages fire a smart-similar burst; every memory cache dies on app restart).
3. On mobile networks, **CGNAT means many users share one IP**, so the 120/min bucket is shared; on Wi-Fi the user has the bucket to themselves. This is exactly why it's "mostly on mobile network."
4. When the Worker returns 429 with `retry-after: <up to 60s>`, the client **retries after 200ms/600ms**, ignoring `Retry-After` — each retry increments the rate counter further, deepening the hole.
5. The UI then **renders network failure as "No results found"** (search) or a dead screen (details), with no error state and no retry affordance. The user waits ~a minute (the rate window resets) and it "magically works."

Fixing this chain (client retry behavior + request-volume reduction + Worker limiter design + honest error states) is Phase 1–3 of the plan and breaks nothing.

## Part 1 — Root cause: mobile-network search/load failures

### The delivery chain
App → axios `tmdbClient` (12s timeout, 2 retries @ 200/600ms) → `streambox-tmdb-proxy.workers.dev` (per-IP limiter 120/min, edge cache 6h/5min-search) → TMDB. Side channels: `api.imdbapi.dev` (ratings), Supabase Edge `external-ratings` (detail only), IMDb/Letterboxd HTML scraping, TVMaze, and the HdFilm/Dizipal scrapers in `WebPlayerService.ts` for playback.

### RC-1: Worker rate limiter design (`workers/tmdb-proxy/src/index.js:68`)
- **120 req/min keyed by `cf-connecting-ip`**. Mobile carriers use CGNAT: hundreds of subscribers share one egress IP. Any two StreamBox users on the same carrier IP share one bucket. One user's cold start can starve another's search.
- The counter is stored in **`caches.default`** — the Cache API is per-datacenter, **non-atomic** (read-increment-write races undercount under parallel bursts), and evictable at any time. It is not a real rate limiter; it's an approximate one that fails unpredictably.
- Rate-limit check runs **before** cache lookup, so even edge-cache **hits burn rate budget**. A fully cached home screen still spends ~40 tokens.
- The 429 response includes `retry-after` — correct — but the client ignores it (RC-3).

### RC-2: Client request-burst volume
Measured from code (with `EXPO_PUBLIC_ENABLE_LIST_EXTERNAL_RATINGS=0` as in `.env`; if EAS production sets it to `1`, add ~2 requests per list item on top):
- **Home cold start**: 2 trending pages + `getPopular()` resolving up to **36 IMDb→TMDB `/find` calls** (`IMDB_POPULAR_SPOTLIGHT_RESOLVE_LIMIT = 36`, concurrency 6) + 2–4 logo requests per visible hero slide. ≈ **40–45 proxied requests**.
- **Movies tab**: top-new discover page + IMDb Top-250 page = **20 `/find` calls**, and the seeding logic (`seedTopNewMovies`/`seedImdbTopMovies`) fetches **additional pages in parallel** until a minimum pool is met — each page another ~20 calls.
- **Detail screen**: details (+credits+external_ids, good) + trailer + external ratings + quick-similar + **smart-similar** (fetches similar/recommendations/keywords/credits for ~20 candidates) ≈ **30–60 requests per open**.
- **Search**: each 350ms-debounced query = `/search/multi` + actor search; typing one word can fire 3–5 bursts.
- All the per-item caches (`imdbIdCache`, `imdbToTmdbMovieCache`, logos, etc.) are **in-memory LruMaps — lost on every app restart**, so every cold start replays the whole burst.

One user browsing Home → Movies → a detail page within a minute plausibly exceeds 120 requests **alone**. On CGNAT it's shared. Result: 429 storms.

### RC-3: The retry design fights the limiter (`src/api/tmdb.ts:466-504`)
- 429s are retried after **200ms and 600ms** — inside the same 60s window, where they are guaranteed to fail *and* each retry increments the counter. The Worker sends `retry-after`; the client never reads it.
- Verdict: under rate limiting, every user-visible request costs 3 rate tokens and still fails in ~1s. This converts a brief 429 into a ~60s outage, matching "wait some time and try again."

### RC-4: Failures rendered as empty states
- `HomeHeader.tsx:282` — search catch: `setResults([]); setHasSearched(true)` → UI shows **"No results for {query}"** on a 429/timeout. The app lies to the user.
- `SearchResultsScreen.tsx:288` — same pattern: `catch { setResults([]) }`, no error UI, no retry.
- `DiscoveryRail` empty state shows the *search* "no results" string for a failed rail.

### RC-5: All-or-nothing loading
- `HomeScreen.loadDiscovery` uses `Promise.all` over 3 rails — one failed request fails the entire screen (retry loop then waits 1.5/4/8s). `Promise.allSettled` with per-rail rendering would keep 2 of 3 rails alive.

### RC-6: Zero connectivity awareness
- `@react-native-community/netinfo` is not installed. No offline banner, no "waiting for connection" state, no request gating on reachability, no auto-refetch on reconnect. On mobile networks with dead zones this is the difference between "feels broken" and "feels aware."

### RC-7 (secondary): search races and stale responses
- `HomeHeader` debounced search has no cancellation/stale-check: an older slow response can overwrite a newer one (classic out-of-order race, most visible on slow mobile links). No AbortController anywhere in the API layer.

### RC-8 (secondary, playback): scraper providers on carrier IPs
- `WebPlayerService.ts` resolves streams by scraping HdFilm/Dizipal with 5–8s per-request timeouts and a 15s total budget. On high-latency mobile links these timeouts trip more often; additionally, those sites' anti-bot layers (and Cloudflare bot protection) throttle carrier CGNAT IPs far more aggressively than residential Wi-Fi IPs. This explains "movie doesn't load, retry later works" for playback specifically. The 15s budget + one bounded retry is reasonable engineering; the variance is environmental.

### Also worth knowing
- `searchMulti` silently drops results with TMDB rating < 6 (`tmdb.ts:2699`) — a niche/older film with rating 5.9 legitimately returns "no results." Combined with RC-4 the user can't distinguish "filtered out" from "failed."
- The proxy runs on the default `*.workers.dev` subdomain. Some carrier DNS/filtering setups are unreliable with workers.dev; a custom domain (you already own `streamboxapp.stream`) removes that variable and is free.

## Part 2 — Performance audit

### Network/caching (dominant cost)
| Issue | Where | Impact |
|---|---|---|
| All API caches in-memory only; cold start replays everything | `tmdb.ts:373-390`, `imdb.ts:91` | Biggest single perf lever. `external_ids`, `/find`, logos, ratings are stable data — persist them |
| 36 `/find` calls to build the hero | `getPopular` `tmdb.ts:1419` | Delays hero, burns rate budget; the mapping IMDb-id→TMDB-id is immutable and should be persisted or shipped in the popular-movies payload |
| Smart-similar burst (~20 candidates × keywords/credits) on every detail open | `tmdb.ts:1976+` | 30–60 requests; only quick-similar is needed for first paint; smart-similar could be deferred/limited |
| Hub seeding fetches multiple pages in parallel | `MoviesScreen.tsx:315-355` | Multiplies page bursts |
| Letterboxd rating scrape per detail | `tmdb.ts:969` | HTML page download on mobile data for one number; failure-cached only in memory |
| Worker rate check costs a cache round-trip per request and counts cache hits | Worker | Adds latency to every request |

### Rendering
- **Good**: FlashList for rails/grids, `memo`'d `MediaCard` with custom comparator, `removeClippedSubviews`, windowed props on FlatLists, `freezeOnBlur: true` on tabs, expo-image with disk cache + recycling keys via `CachedRemoteImage`, skeletons (`HubSkeletons`), `startTransition` for cache-refresh state swaps.
- **Gaps**:
  - `SpotlightCarousel` uses RN `Animated.Image` for **w780 backdrops — no disk cache**; heroes re-download every cold start (largest images in the app). Also `keyExtractor` by index and 3× tripled data for the infinite loop.
  - `SearchResultsScreen` and `HomeHeader` dropdown use plain RN `Image` for posters — no disk cache on the highest-churn surfaces.
  - `SearchResultsScreen` staggered `FadeInDown.delay(index * 40)` applies to every item, including during pagination — late pages animate in with long cumulative delays.
  - styled-components runtime CSS parsing on every theme change is a minor tax; fine to keep, but don't grow the pattern into hot paths (player controls).

### Startup
- Startup orchestration is genuinely good (font fallback timers, splash-over-content mounting, hydrated hub caches gating first mount, storage migrations, telemetry `app_ready`). No action needed beyond keeping `dist/` (32 MB) and stray data out of the repo/build context.

| Best practice | Status |
|---|---|

| Offline detection + reconnect refetch | ❌ NetInfo not installed |

## Part 5 — Design critique (senior product designer view)

### Overall impression
This is a competent, coherent dark streaming UI that clears the "AI slop" bar: real token system (`Theme.ts` — spacing/radius/motion/shadows), tinted neutrals (`#0D100F` green-tinted, not `#000`), four accent themes, a single distinctive typeface (Outfit) with tabular numerals for ratings, a custom SVG comet loader instead of a stock spinner, parallax hero with animated pagination dots, skeletons, and press-scale micro-interactions. It reads "solid mid-tier commercial app." The gap to top-tier (Netflix/Apple TV class) is not visual style — it's **state honesty, transitions, and texture**.

### Nielsen heuristics scorecard
| # | Heuristic | Score /4 | Key issue |
|---|---|---|---|
| 1 | Visibility of system status | 2 | Failures render as empty states; no offline indicator; silent background refreshes |
| 2 | Match with real world | 3 | Good genre chips/labels; "No results" shown for filtered-out or failed queries is a mismatch |
| 3 | User control & freedom | 3 | Good back behavior; carousel auto-advance can't be paused; no undo on watch-state toggles |
| 4 | Consistency & standards | 2.5 | Typography drift (system bold vs Outfit); RN Image vs CachedRemoteImage; two search result surfaces styled differently |
| 5 | Error prevention | 3 | Quality gate on items, quality warning modal — good |
| 6 | Recognition over recall | 3 | No recent searches; filters reset per session |
| 7 | Flexibility & efficiency | 2.5 | No long-press quick actions on posters (add to watchlist without opening detail) |
| 8 | Aesthetic & minimalist | 3 | Clean, but every rail is the same 132×198 card; rating badge on **every** poster is noise |
| 9 | Error recovery | 1.5 | No retry buttons anywhere outside startup boundary; the #1 gap |
| 10 | Help & documentation | 3 | Mostly self-evident UI |
| | **Total** | **26.5/40** | mid-tier; recovery + status honesty are the drag |

### Priority design faults
1. **[P0] Error ≠ Empty.** "No results found" on a 429 destroys trust in search — the user's own report proves it ("doesn't return anything… try again and it works"). Every fetch surface needs three distinct states: error (with Retry button), empty (with guidance), loaded.
2. **[P0] No offline/poor-connection acknowledgment.** A tiny animated banner ("You're offline — showing saved content") + auto-retry on reconnect makes slow networks feel handled instead of broken.
3. **[P1] Layout monotony.** Home is hero + three identical card rails. Top-tier apps vary the rhythm: one ranked rail (big numerals), one landscape-backdrop rail, one portrait rail. You already have `rank` data from Top-250 — a "Top 10 this week" rail with oversized numbers is nearly free and adds identity.
4. **[P1] Rating badge on every poster** = visual static. Show it on focus/press, or only above a threshold (e.g., 8.0+ gold badge), letting posters breathe.
5. **[P2] Transition flatness.** No shared-element transition from poster → detail; detail hero is static. Poster-to-hero shared transitions + a scroll-collapsing detail header are the single biggest "feels premium" upgrade available. (Reanimated 4 + native-stack supports this well now.)
6. **[P2] Typography drift** (system bold in SearchResults/etc.). One pass to enforce `Outfit_*` + `theme.typography` everywhere.
7. **[P3] Carousel details**: auto-advance doesn't pause while the user is pressing/reading; dots + `keyExtractor` index on tripled data; consider progress-bar dots (fill over 5s) so the advance is predictable.

### On animation tooling (your Remotion/Rive question)
- **Remotion is not applicable** — it renders videos in React on servers/CLI, not app UI. Skip it.
- **Reanimated 4 (already installed) is the correct primary tool** — you're using maybe 20% of it. Highest-ROI additions: shared element transitions, scroll-driven detail header, skeleton shimmer (looping gradient), staggered rail entrance on first load only, press haptics (`expo-haptics`, ~1 line per press).
- **Rive** (~400KB) is worth adding for exactly three things: an animated empty/error mascot state, a badge-unlock celebration, and an animated splash/logo. Interactive state machines, tiny files. Use sparingly — brand moments, not chrome.
- **Lottie** is the simpler alternative if you just want canned empty-state animations; pick one of Rive/Lottie, not both.
- **Skia** only if you later want fancy stats charts or blur-gradient posters; not needed now.

## Part 6 — Free-tier posture (Cloudflare + Supabase)

- Cloudflare Workers free: 100k req/day. Your problem is not the daily cap — it's the self-imposed 120/min/IP limiter plus request volume. Reducing client volume (Phase 3) is worth ~5–10× headroom; the limiter redesign (Phase 2) removes the CGNAT failure mode.
- The Worker's edge cache only stores 2xx (good), TTLs 6h/5min are sane; add `stale-while-revalidate` semantics and cache 404s for ~5min to stop repeat misses from burning upstream calls.
- Supabase free: correctly kept off the browsing hot path (`external-ratings` detail-only, flag-gated lists per `docs/backend-hybrid-architecture.md`). Telemetry writes should be verified batched (`flushTelemetry` exists) so `app_telemetry_events` doesn't chew the free row/egress budget.
- `head_sampling_rate: 0.1` on Worker observability means you're sampling 10% — enough to confirm this audit: check the dashboard for `tmdb_proxy_rate_limited` events; likewise `select * from app_telemetry_events where event_category='tmdb' and metadata->>'status'='429'`.

## Part 7 — Implementation plan (phased, zero-breakage)

Rules baked into every phase: additive changes behind existing seams; `npm run typecheck && npm test` after each step (architecture tests protect the player); ship each phase as its own OTA to the correct runtime track; verify `runtimeVersion` before every `eas update` (see docs/release-tracks.md).

### Phase 0 — Confirm with data (half a day, no code)
1. Cloudflare dashboard → Worker logs: count `tmdb_proxy_rate_limited` by hour; correlate with user-reported failure times.
2. Supabase: query `app_telemetry_events` for `tmdb` 429s and `durationMs` distributions (you already track `rateLimitRemaining` — it will show buckets draining).
3. Check EAS production env: whether `EXPO_PUBLIC_ENABLE_LIST_EXTERNAL_RATINGS` is `1` in prod (if yes, list enrichment multiplies every burst and Phase 3 matters even more).

### Phase 1 — Stop the bleeding, client-side (1–2 days, low risk)
1. **Honor `Retry-After`** in the axios response interceptor: on 429, if `retry-after` ≤ ~8s, wait that long once; if longer, fail fast to the UI error state. Never blind-retry a 429 at 200ms.
2. **Real error states**: search dropdown, SearchResultsScreen, DiscoveryRail — distinguish catch-path (error + Retry button) from genuinely empty results. Localize via existing i18n resources.
3. **`Promise.allSettled`** in `HomeScreen.loadDiscovery` and the Movies/Series hub loaders: render whatever succeeded, retry only what failed.
4. **Stale-response guard** in `HomeHeader` search (compare query at resolve time; or AbortController per keystroke).
5. Add a "couldn't refresh — showing saved content" toast when a background refresh fails but cache exists (SWR honesty).

### Phase 2 — Worker fixes (1 day, isolated deploy, instantly reversible)
1. Reorder: **cache lookup before rate-limit check** — cache hits become free.
2. Replace the per-IP Cache-API counter: simplest correct option is to drop app-level limiting to a high safety ceiling (e.g., 600/min/IP) purely as abuse protection and let TMDB's own limits govern; the proper option is a Durable Object or KV token bucket keyed by a client-generated `x-streambox-install-id` header (per-device fairness immune to CGNAT). Keep the header optional so old clients keep working.
3. Cache 404s for 5 min; add `stale-while-revalidate: 86400` to cached responses.
4. Route the Worker through a custom domain (`tmdb.streamboxapp.stream`) to bypass workers.dev quirks on carrier networks.

### Phase 3 — Request-volume diet (2–4 days, medium)
1. **Persist the immutable maps** (TMDB↔IMDb id maps, logo paths, imdb ratings with 24h TTL) through the existing `runtimeCache` persistent layer instead of pure in-memory LruMaps. Cold-start bursts collapse to near zero on second launch.
2. Cap `getPopular` spotlight resolution to ~12 (you display 10) instead of 36, resolving the rest lazily.
3. Defer `getSmartSimilarMovies/Series` until the user scrolls near the similar rail (quick-similar already paints it).
4. Make hub "seed extra pages" sequential-on-demand rather than parallel-upfront.
5. Optional: Worker batch endpoint (`/batch/external_ids?ids=…`) later; not required once 1–4 land.

### Phase 4 — Connectivity awareness (1–2 days)
1. Add `@react-native-community/netinfo`; expose `useConnectivity()`.
2. Offline banner component; gate background refreshes offline; auto-refetch hubs/search on reconnect.
3. Optional low-bandwidth mode: poster `w185` + skip hero logos when `details.isConnectionExpensive || cellularGeneration ≤ 3g`.

### Phase 5 — Design elevation (1–2 weeks, incremental, purely additive)
1. State-trio components (Error/Empty/Offline) with one Rive or Lottie illustration set.
2. Typography enforcement pass (kill `font-weight` without family); swap SearchResults/HomeHeader/Spotlight images to `CachedRemoteImage`.
3. Shared-element poster→detail transition + scroll-collapsing detail hero (Reanimated 4).
4. Haptics on primary presses; skeleton shimmer; pause carousel while pressed; progress-fill dots.
5. "Top 10" ranked rail with oversized numerals to break rail monotony; rating badge shown selectively.

### Phase 6 — Code health (ongoing, opportunistic)
1. Extract shared `MediaHubScreen` from Movies/Series screens (config-driven rails) — do it during Phase 5 while touching both.
2. Split `tmdb.ts` (client/interceptors, lists, details, similar-ranking, scrapers) behind an unchanged barrel export so no imports break.
3. Repo hygiene: gitignore/remove `dist/`, `outputs/`, `list/`, `letterboxd-*/`, `scripts/__pycache__`; move root prompt `.md`s into `docs/`.
4. CI: run `typecheck`, `lint`, `test` on every PR if not already.

### Success metrics
- `tmdb_proxy_rate_limited` events → ~0.
- Search failure→"empty" misreports → 0 (new telemetry event `search_error_shown`).
- Cold-start proxied request count: from ~40–120 to <15 (first launch) and <5 (warm).
- `app_ready` p95 unchanged or better.
